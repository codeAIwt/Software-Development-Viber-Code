# 软件工程实现：产品需求说明（PRD）
> 本文档的最终阅读者是 AI（如 Cursor、Windsurf、Claude）。

# 一：全局上下文与架构约束 (Context & Constraints)
- 采用B/S架构
- 前端框架：采用Vue框架
- 后端框架：采用VScode的python3.12.5编程环境
- 获取数据：WebSocket
- 数据库：MySQL
- 强制规范：
  1. 所有用户输入需做前端拦截校验+后端二次校验，避免非法数据入库
  2. 敏感数据（如密码）需通过不可逆加密算法（如bcrypt）处理后存储
  3. 接口请求需携带Token鉴权，无有效Token拒绝核心接口访问
  4. 定时任务执行结果需记录日志，异常时触发告警机制
  5. 视频流处理仅在本地完成，禁止传输原始未处理画面至服务端

# 二：核心功能（MVP功能）与操作流程 (Core Workflows) 【核心业务说明】
## 核心功能1：用户系统
- 页面交互与执行步骤(Step-by-Step Flow)：
1. 未登录用户进入平台任意页面，均会触发轻量弹窗提示，提供 “登录”“注册” 快捷入口，也可关闭弹窗进入游客模式（仅可浏览，无法使用自习室、时长统计等核心功能）。
2. 注册流程： 
    1. 用户点击 “注册” 进入 /register 路由页面，页面展示注册表单，包含：手机号（文本，唯一）、密码（文本，6-16 位字母数字组合）、确认密码（文本）；
    2. 用户填写所有信息后点击 “完成注册”，系统拦截校验：两次密码是否一致、密码是否符合格式；
    3. 数据操作：调用后端 API，向用户表（user）插入一条新记录，包含手机号、加密后的密码、注册时间，默认用户昵称设为 “自习小伙伴_+ 随机数字”，头像设为平台默认图；
    4. 页面反馈：注册成功后，Toast 提示 “注册成功，自动为您登录”，自动完成登录并跳转到平台首页 /home。
3. 登录流程：
    1. 用户点击 “登录” 进入 /login 路由页面，提供并展示 “手机号 + 密码”表单，包含：手机号、密码
    2. 填写信息后点击“立即登录”，系统拦截校验表单信息是否为空，随后调用后端 API 校验账号信息有效性；
    3. 数据操作：校验通过后，后端生成登录令牌（Token），关联用户 id 并设置有效期，前端将 Token 存储在本地缓存，同时拉取用户基础信息、学习数据；
    4. 页面反馈：登录成功后，Toast 提示 “登录成功”，跳转到平台首页 /home；校验失败Toast 提示“登录失败，请重试”。
4. 退出登录流程：
    1. 已登录用户在个人中心 /personal 页面，点击 “退出登录” 按钮，弹出确认弹窗 “确定要退出当前账号吗？”；
    2. 用户点击 “确认”，系统执行数据操作：前端清除本地缓存的登录 Token，后端销毁当前用户的登录状态；
    3. 页面反馈：退出成功后，Toast 提示 “已成功退出登录”，自动跳转到平台登录页 /login。

## 核心功能2：全应用自习时长激励排行榜
- 页面交互与执行步骤(Step-by-Step Flow)：
1. 用户进入个人中心 /personal 路由页面，找到 “学习数据” 模块。
2. 页面自动拉取并展示用户当日自习累计时长、日平均自习时长，以及核心展示项 “击败 XX% 全平台自习成员(调用计算函数，计算方式：按每小时分段，并统计全平台不同时间段的人数，然后统计“低于用户所在时间段的人数和”/“全平台用户数” *100%，作为展示数据)”。
3. 系统每日凌晨 4 点触发定时任务，统计全应用用户前一日的自习时长并计算平均值，基于用户个人数据匹配对应的击败百分比。
4. 数据操作：后端通过定时任务遍历用户学习时长表，计算全量用户数据后，将用户个人日平均时长、击败百分比同步至 Redis 缓存，前端从缓存拉取数据展示。
5. 页面反馈：若用户当日无自习记录，展示 “今日尚未开始学习，快去开启自习吧” 提示语，无当日击败百分比数据；有数据时实时展示统计结果，数据更新后页面自动刷新。

## 核心功能3：学习主题匹配及自习室创建 / 加入
- 页面交互与执行步骤(Step-by-Step Flow)：
1. 用户进入自习室主页面 /study-room 路由页面，可选择 “创建自习室” 或 “加入自习室” 按钮。
2. 若选择创建自习室：
    1. 弹出创建表单，包含学习主题选择（下拉框，如考研、备战期末、考公、语言学习等）、自习室最大人数设置（数字输入，默认 4 人，上限 8 人）；
    2. 用户确认创建后，系统校验主题是否选择、人数是否在合理区间；
    3. 数据操作：调用后端 API，向自习室表插入记录，状态设为 “空闲”，关联创建者 user_id 与选定学习主题标签；
    4. 系统自动从同主题的空闲用户队列中匹配对应人数的用户，推送自习室加入邀请。
3. 若选择加入自习室：
    1. 页面展示所有 “空闲” 状态的自习室列表，包含自习室主题、当前人数 / 最大人数、创建时间；
    2. 用户点击目标自习室的 “加入” 按钮，系统校验该自习室是否仍有空闲位置；
    3. 数据操作：调用后端 API，将用户 user_id 关联至该自习室表，更新自习室当前人数；
4. 页面反馈：创建 / 加入成功后，全局 Toast 提示 “操作成功”，并重定向至该自习室的在线伴学页面 /study-room/[id]；若校验失败（如人数超限、无空闲位置），Toast 提示“加入失败，请重试”。

## 核心功能4：隐私化在线伴学
- 页面交互与执行步骤(Step-by-Step Flow)：
1. 用户进入自习室在线伴学页面 /study-room/[id]，页面加载完成后弹出权限申请弹窗，申请摄像头、麦克风使用权限。
2. 用户授权后，系统默认执行：开启摄像头采集画面，关闭麦克风音频传输，将用户学习画面展示在自习室的成员视频列表中。
3. 页面提供隐私模式切换入口，位于视频画面角落，包含 “原画面”“仅展示手部”“背景模糊” 三个选项。
4. 若用户切换隐私模式：
    1. 选择 “仅展示手部”：前端对摄像头采集的视频流进行画面裁剪，仅保留手部区域展示；
    2. 选择 “背景模糊”：前端调用滤镜接口，对视频流背景进行高斯模糊处理，保留人物主体清晰；
5. 数据操作：用户的模式选择同步至后端，在自习室成员列表中标注该用户的展示模式，视频流处理均在本地完成，不传输原始未处理画面。
6. 页面反馈：权限授权成功后，视频画面正常加载；授权失败时，Toast 提示 “请开启摄像头权限以进行在线伴学”；模式切换成功后，视频画面实时更新，无额外弹窗干扰；用户可随时点击摄像头关闭按钮，关闭后视频画面展示为 “已关闭摄像头” 占位图。

# 四：核心状态机 (State Machine)
## 加入自习室状态机
1. 初始状态：未加入
   - 触发事件：用户点击“加入自习室”按钮
   - 校验规则：自习室状态为“空闲”且当前人数 < 最大人数
   - 状态转换：未加入 → 加入中
2. 加入中状态
   - 成功事件：后端完成用户与自习室关联、更新当前人数
   - 状态转换：加入中 → 已加入
   - 页面反馈：Toast提示“加入成功”，跳转至自习室伴学页面
   - 失败事件：校验不通过（人数满/自习室关闭）
   - 状态转换：加入中 → 未加入
   - 页面反馈：Toast提示“加入失败，请重试”
3. 已加入状态
   - 触发事件1：用户主动离开自习室
   - 状态转换：已加入 → 已离开
   - 数据操作：更新leave_time，自习室current_people减1
   - 触发事件2：自习室被关闭
   - 状态转换：已加入 → 已离开
   - 数据操作：更新leave_time，标记自习室状态为closed

## 创建自习室状态机
1. 初始状态：未创建
   - 触发事件：用户点击“创建自习室”按钮并提交表单
   - 校验规则：主题已选择、最大人数在1-8区间
   - 状态转换：未创建 → 创建中
2. 创建中状态
   - 成功事件：后端完成自习室记录插入、状态设为“空闲”
   - 状态转换：创建中 → 已创建
   - 数据操作：自动匹配同主题空闲用户并推送邀请
   - 页面反馈：Toast提示“创建成功”，跳转至自习室伴学页面
   - 失败事件：表单校验失败/数据库插入异常
   - 状态转换：创建中 → 未创建
   - 页面反馈：Toast提示“创建失败，请重试”
3. 已创建状态
   - 触发事件1：自习室人数达到最大值
   - 状态转换：已创建 → 已满员
   - 数据操作：更新自习室status为full
   - 触发事件2：创建者主动关闭自习室
   - 状态转换：已创建 → 已关闭
   - 数据操作：更新自习室status为closed、close_time为当前时间
   - 触发事件3：无成员活动超2小时（定时检测）
   - 状态转换：已创建 → 已关闭
   - 数据操作：更新自习室status为closed、close_time为当前时间

## 自习室状态流转总机
1. 空闲（idle）：自习室已创建且有空闲位置
   - 触发事件：成员加入至人数满 → 转换为满员（full）
   - 触发事件：创建者关闭/超时无活动 → 转换为关闭（closed）
2. 满员（full）：自习室人数达到最大值
   - 触发事件：创建者关闭/超时无活动 → 转换为关闭（closed）
   - 触发事件：有成员离开且人数<最大值 → 转换为空闲（idle）
3. 关闭（closed）：自习室停止使用
   - 触发事件：无（终端状态）

## 用户视频隐私模式状态机
1. 初始状态：原画面（original）
   - 触发事件1：用户选择“仅展示手部”
   - 状态转换：original → hand_only
   - 数据操作：更新study_room_member表video_mode字段，本地裁剪视频流
   - 触发事件2：用户选择“背景模糊”
   - 状态转换：original → blur_bg
   - 数据操作：更新study_room_member表video_mode字段，本地处理视频流高斯模糊
2. hand_only/blur_bg状态
   - 触发事件：用户切回“原画面”
   - 状态转换：hand_only/blur_bg → original
   - 数据操作：更新video_mode字段，恢复原始视频流展示
   - 触发事件：用户切换至另一隐私模式
   - 状态转换：hand_only ↔ blur_bg
   - 数据操作：更新video_mode字段，切换对应视频处理逻辑

# 五：防御性边界与异常流 (Edge Cases)
## 5.1 权限与越权防御
### 5.1.1 登录态与接口鉴权
1. 未登录用户访问核心接口（/api/study-room/create、/api/study-duration/query等）：
   - 后端拦截：返回401 Unauthorized，携带“请先登录”提示
   - 前端处理：自动跳转至/login页面，Toast提示“登录状态已失效，请重新登录”
2. 越权访问他人数据：
   - 场景1：用户尝试通过修改请求参数（如user_id）查询/修改其他用户的学习时长
   - 防御逻辑：后端接口校验请求Token关联的user_id与请求参数user_id是否一致，不一致则返回403 Forbidden
   - 场景2：用户尝试加入非空闲状态的自习室/修改他人创建的自习室信息
   - 防御逻辑：接口层校验自习室状态、用户与自习室的关联关系，非创建者禁止修改自习室核心信息
3. 权限升级攻击：
   - 防御逻辑：所有接口操作记录用户IP、操作时间，高频异常操作（如1分钟内10次以上鉴权失败）触发IP临时封禁（10分钟），并记录风控日志

### 5.1.2 隐私权限防御
1. 摄像头/麦克风权限拒绝：
   - 前端处理：弹窗提示“未获取摄像头权限，无法参与在线伴学，是否重新申请？”，提供“重新申请”/“取消”按钮；取消后展示“未开启摄像头”占位图，仍可留在自习室但无视频展示
2. 视频流传输防御：
   - 强制规则：所有视频流处理均在本地完成，服务端仅传输用户隐私模式标记，不接收任何原始视频数据
   - 前端防御：禁用视频流截图、录屏检测（可选），提示用户“请勿泄露个人隐私画面”

## 5.2 并发与数据异常防御
### 5.2.1 并发操作防御
1. 多人同时加入同一自习室：
   - 场景：自习室剩余1个位置，2名用户同时点击“加入”
   - 防御逻辑：后端接口加分布式锁（Redis锁），仅允许第一个请求成功，第二个请求返回“自习室已无空闲位置”，前端Toast提示失败
2. 并发创建自习室：
   - 防御逻辑：用户1分钟内仅允许创建1个自习室，通过Redis计数限制，避免高频创建垃圾自习室
3. 定时任务并发执行：
   - 场景：每日凌晨4点的时长统计任务被重复触发
   - 防御逻辑：定时任务执行前获取分布式锁，锁释放前其他任务实例无法执行，避免重复计算数据

### 5.2.2 数据异常防御
1. 学习时长数据异常：
   - 场景1：用户当日自习时长超过24小时（数据录入错误/恶意篡改）
   - 防御逻辑：后端入库前校验total_duration ≤ 1440（分钟），超出则按1440处理并记录异常日志
   - 场景2：用户学习时长表中同一user_id+study_date存在多条记录
   - 防御逻辑：入库前先查询是否已有当日记录，存在则执行更新操作（累加时长），而非插入，避免数据重复
2. 自习室人数异常：
   - 场景：current_people > max_people
   - 防御逻辑：每次更新current_people前校验，若更新后超过max_people则拒绝操作，返回“人数超限”异常，并自动触发自习室状态校验（标记为full）
3. 缓存与数据库数据不一致：
   - 场景：Redis缓存的击败百分比与数据库计算结果不一致
   - 防御逻辑：定时任务执行完成后，对比缓存与数据库核心数据，差异率>1%时触发缓存全量刷新；前端拉取缓存失败时，降级从数据库直接查询
4. 空数据/NULL值防御：
   - 防御逻辑：所有核心字段（如user表的phone、password，study_room表的theme）入库前校验非空，前端表单拦截空值提交，后端接口二次校验，避免NULL值导致的业务异常
