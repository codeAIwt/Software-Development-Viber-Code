## 5. 错误分析

### 5.1 错误类型分布

| 错误类型         | HTTP状态码 | 错误描述        | 数量   | 占比  |
| ------------ | ------- | ----------- | ---- | --- |
| Rate Limit   | 429     | 请求过于频繁      | 430+ | 39% |
| Unauthorized | 401     | Token无效或已过期 | 394  | 36% |
| Not Found    | 404     | 资源不存在       | 47   | 4%  |
| Bad Request  | 400     | 请求参数错误      | 34   | 3%  |
| 其他           | -       | 其他错误        | 195  | 18% |

### 5.2 错误详细分析

#### 5.2.1 Rate Limit (429) - 限流触发

**问题描述**：

- 注册接口限流：每IP最多10次/分钟
- 登录接口限流：每IP最多20次/分钟

**错误日志**：

```
POST /api/user/register: HTTPError('429 Client Error: Too Many Requests')
POST /api/user/login: HTTPError('429 Client Error: Too Many Requests')
```

**分析结论**：

- ✅ **这是正常行为，不是Bug**
- 限流保护机制正常工作
- 防止暴力破解和滥用

#### 5.2.2 Unauthorized (401) - 认证失败

**问题描述**：

- Token无效或已过期
- 用户登出后仍尝试访问需认证的接口

**错误日志**：

```
GET /api/user/profile: HTTPError('401 Client Error: Unauthorized')
PUT /api/user/profile/nickname: HTTPError('401 Client Error: Unauthorized')
PUT /api/user/profile/tags: HTTPError('401 Client Error: Unauthorized')
```

**分析结论**：

- ✅ **这是测试场景问题，不是代码Bug**
- 用户登出后Token失效
- 测试代码在登出后仍尝试使用该Token访问受保护资源
- 实际生产环境中用户不会在登出后继续使用旧Token

#### 5.2.3 Not Found (404) - 资源不存在

**问题描述**：

- 查询自习室信息时房间已被销毁
- 查询的房间ID不存在

**错误日志**：

```
GET /api/room/info/{room_id}: HTTPError('404 Client Error: Not Found')
DELETE /api/room/destroy/{room_id}: HTTPError('404 Client Error: Not Found')
```

**分析结论**：

- ✅ **这是并发测试的正常现象**
- 多用户并发操作导致房间被其他用户销毁
- 属于正常的业务逻辑处理

#### 5.2.4 Bad Request (400) - 请求错误

**问题描述**：

- 请求参数不符合要求

**错误日志**：

```
POST /api/room/create: HTTPError('400 Client Error: Bad Request')
POST /api/room/join: HTTPError('400 Client Error: Bad Request')
POST /api/user/login: HTTPError('400 Client Error: Bad Request')
```

**分析结论**：

- ✅ **这是测试数据问题**
- 登录时使用未注册的手机号
- 或用户已在其他房间尝试创建新房间

## 7. 性能瓶颈与优化建议

### 7.1 性能瓶颈

| 瓶颈点       | 描述                   | 严重程度 | 建议             |
| --------- | -------------------- | ---- | -------------- |
| 注册接口      | 95%ile达到130ms        | 中    | 考虑优化密码哈希或数据库写入 |
| Redis锁竞争  | 高并发时join/leave操作延迟增加 | 低    | 考虑优化锁粒度        |
| fakeredis | 测试环境使用模拟Redis        | N/A  | 生产环境使用真实Redis  |

### 7.2 优化建议

1. **数据库优化**
   - 考虑添加数据库索引
   - 使用连接池管理数据库连接
2. **缓存优化**
   - 生产环境使用真实Redis替代fakeredis
   - 优化缓存键设计减少冲突
3. **限流优化**
   - 当前限流基于IP，可考虑结合用户ID
   - 添加更细粒度的限流策略
4. **异步处理**
   - 对于非关键操作考虑异步处理
   - 减少用户等待时间
